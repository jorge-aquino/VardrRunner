# v0.21.0 ‚Äî Run-scoped pipeline isolation

**Date:** 2026-06-20

Feature release. No breaking changes to the CLI or public API.

## Problem

Before this release, every pipeline stage with `source="recon"` read targets from the
shared backend recon store with no filtering. If an operator had run subfinder the day
before, its hosts were already in the store. A new pipeline run would silently pick them
up in addition to what subfinder just found ‚Äî mixing old and new recon with no way to
tell them apart. A live `subfinder ‚Üí httpx ‚Üí nuclei` run with 736 pre-existing hosts
would hand 736 + N hosts to nuclei instead of just the N freshly discovered ones.

## Solution

### Local handoff files

After every successful pipeline stage, the handler extracts the targets it produced and
writes them to `run_dir/handoff.txt`. The next stage reads from that file instead of
calling `client.recon()`. Targets are isolated to the run; no prior data bleeds in.

### Run ID

`pipeline run` generates a short `uuid4().hex[:8]` run ID at the start and prints it at
the end. Operators can use it to cross-reference logs. It is the seed for future backend
run-scoped filtering once VardrMap supports it.

### Fallback

If `extract_handoff_targets()` returns `[]` (terminal tools: nuclei, nmap, naabu; or an
unreadable output file), the next stage falls back to `client.recon()`. This preserves
the original behaviour for mixed pipelines and for standalone `run` commands.

## New handler methods

- `extract_handoff_targets(output: Path) -> list[str]` ‚Äî extracts stage output into a
  flat target list. Implemented for `HttpxHandler` (urls/hosts from JSONL),
  `SubfinderHandler` and `DnsxHandler` (hosts from JSONL). Returns `[]` by default
  (all other handlers).
- `normalize_handoff_targets(targets: list[str]) -> list[str]` ‚Äî post-processes targets
  from a handoff file before passing them to `execute()`. `NmapHandler`, `DnsxHandler`,
  and `NaabuHandler` override this to strip URL scheme/path to bare host, matching what
  their `resolve_targets()` does for backend recon. Default is identity.

## Changed internals

`_run_stage()` in `commands/pipeline.py` changed return type from `bool` to
`tuple[bool, Path | None]`. This is an internal function ‚Äî no user-facing change.
