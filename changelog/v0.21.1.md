# v0.21.1 — Quick fixes

**Date:** 2026-06-20

Patch release. No breaking changes.

## Fixes

### `import` command — SUPPORTED_TOOLS corrected

`SUPPORTED_TOOLS` was `["nuclei", "httpx", "ffuf"]`. `ffuf` has no handler anywhere
in the codebase and the backend has no `ffuf` import type — it would always fail at
runtime with a 422 or 400. It has been removed.

The correct list is `["httpx", "nuclei"]`:
- `subfinder`/`dnsx` are excluded because their upload method converts output to
  httpx-format JSONL and calls `import_file(..., "httpx", ...)`. A user importing a
  raw subfinder `.txt` file would need to use tool type `httpx` after converting it.
- `nmap`/`naabu` use `client.create_services()` (a structured JSON payload), not the
  file-import endpoint.

### Run directory pruning

`_make_run_dir()` in `commands/run.py` creates a timestamped subdirectory under
`~/.vardrmap/runs/` for every tool run and pipeline stage. On a VPS running daily
pipelines with 4-stage `deep` runs, that's 28+ directories per week with no cleanup.

`_prune_run_dirs()` now runs on every `_make_run_dir()` call, deleting any directory
in `~/.vardrmap/runs/` whose mtime is older than 7 days. The prune is a no-op if the
runs directory doesn't exist yet, and uses `shutil.rmtree(ignore_errors=True)` so a
locked or permission-denied directory doesn't fail the main operation.

## UX

### `pipeline list` shows stage sources

Before: `recon: subfinder → httpx → nuclei`
After:  `recon: subfinder(scope) → httpx(recon) → nuclei(recon)`

Operators can now see at a glance where each stage reads its targets from without
reading the source code or running with `--dry-run`.
