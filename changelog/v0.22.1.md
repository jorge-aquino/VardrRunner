# v0.22.1 — Scope safety: --max-targets guardrail

**Date:** 2026-06-20

Minor release. No breaking changes; default behaviour is unchanged for targets ≤ 500.

## Problem

A wildcard scope like `*.example.com` with many subdomains could cause subfinder to
pass thousands of hosts to httpx or nuclei with no warning. With `--yes` set (common
in scripts and daemon pipelines), the operator had no opportunity to review or cancel.

## Solution

`--max-targets N` (default: 500) on every `run <tool>` command and on `pipeline run`.
When the resolved target count exceeds the limit:

1. The command prints the actual count and the limit.
2. It aborts **before running any tool** — no partial scan, no upload.
3. It tells the operator how to raise (`--max-targets N`) or disable (`--max-targets 0`) the cap.

The check fires even when `--yes` is set, because `--yes` means "skip the confirmation
prompt" — not "ignore safety guards."

## Specifics

- `run.py`: `MAX_TARGETS_DEFAULT = 500`, `_check_target_cap(targets, max_targets)`. All
  six `run_*` functions (`httpx`, `subfinder`, `nuclei`, `nmap`, `dnsx`, `naabu`) call it
  after target resolution and before `_confirm`.
- `pipeline.py`: `run_pipeline` and `_run_stage` both accept `max_targets`. The check
  runs in `_run_stage` after target resolution (including handoff-file reads). Each stage
  is checked independently, so a pipeline that was under the limit at stage 1 can still
  trip the cap at stage 2 if the previous stage produced more targets than expected.
